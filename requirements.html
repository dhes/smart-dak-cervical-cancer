Component 9 of the WHO SMART Guidelines L2 DAK structure. This page enumerates the high-level functional and non-functional requirements that a digital system must satisfy in order to implement this DAK — that is, the software capabilities that would let a country programme actually run Algorithm 5 with clinical decision support, scheduling, and indicator reporting in place.

Per Section 3.1 of the SOP, Component 9 is "a high-level list of core functions and capabilities that the system must have to meet the end users' needs and achieve tasks within the business process." Its purpose is "system design to know what the system should be able to do."

Important framing. Component 9 is about the target digital system that implements this DAK — not about the DAK itself. If a country decides to deploy this DAK, their IT team (or their software vendor) needs a specification they can build or procure against. Without Component 9, that team has to derive the system specification themselves by reading Components 1–8, with predictable drift and omissions. Component 9 does that derivation work once, for them.

Scope and authoring posture

Algorithm-5 scope. Consistent with the discipline of the preceding components, these requirements reference Algorithm 5 cascade vocabulary. Capabilities that would be required for a different algorithm family (dual-stain cytology module, HPV mRNA primary platform, cytology lab integration for cytology-primary programmes) are out of scope — a programme running a different algorithm would author its own Component 9.

Phase-1 derivation. The requirements below are analytically derived from Components 1–8. No implementer-panel or vendor consultation has occurred. SME validation for this component would ideally come from a national or regional digital health architect with experience implementing WHO SMART DAKs; the same stratified deferral pattern as earlier components applies.

Cross-reference approach. Each requirement carries a pointer to the originating component(s). Component 9 is in substantial part a restatement of Components 1–8 in "the system SHALL…" language pivoted to the implementer reader. That restatement is not redundancy — it's the delivery form the SOP asks for.

Conventions.

  • SHALL denotes a hard requirement — an implementation that does not satisfy the requirement is non-conformant to this DAK.
  • SHOULD denotes a strong preference with room for justified deviation in a particular deployment.
  • MAY denotes an optional capability.

Requirement identifiers use FR-NN for functional and NFR-NN for non-functional. Numbering is sequential within each category and stable — a given ID refers to the same capability across all versions of this page.

Functional requirements

Twenty functional requirements, organised by the cascade module they support, plus a cross-cutting cluster for capabilities that underwrite the whole cascade.

Cross-cutting capabilities

These requirements are foundational — every downstream module depends on them.

FR-01 The system SHALL assign or consume a stable patient identifier that uniquely identifies a woman across all her encounters in the cascade, and SHALL persist that identifier across device replacement, facility transfer, and client-app reinstallation. All cascade personas Without a stable patient identifier, Tier-B indicators (IND-04 through IND-08), cascade-attrition tracking, and cross-visit scheduling are not instrumentable. SPI is the foundation on which almost every other requirement rests. cc-000; Indicators — Tier A/B; Scheduling Logic; NFR-06
FR-02 The system SHALL evaluate the decision tables published under Decision Logic given the current cascade state of a given woman, and SHALL return the decision output (e.g., due-for-screening, triage-disposition, treatment-eligibility) at the point of care in time to inform clinician action. Nurse/midwife; Advanced clinician The decision logic is this DAK's principal clinical artifact; a system that cannot evaluate it cannot provide the decision support Algorithm 5 requires. Component 6; NFR-08
FR-03 The system SHALL capture every data element listed in the Core Data Dictionary, in the format and with the value set defined there, at the cascade step where the CDD and the BPMN workflows indicate the element is collected. All data-entry personas (CHW, Nurse/midwife, Laboratory technician, Advanced clinician) Indicators, decision logic, and cascade-state tracking all depend on CDD elements being present and well-formed. Capture is the foundation beneath all derived capability. Component 5; Component 4
FR-04 The system SHALL maintain, for each woman, her current cascade state — at minimum: eligible-not-screened, screened-HPV-negative, screened-HPV-positive-awaiting-triage, triage-negative, triage-positive-awaiting-treatment, treated-awaiting-retest, referred-for-suspected-cancer, treatment-deferred-pregnancy, lost-to-follow-up — and SHALL update state on every cascade-affecting event. Nurse/midwife; Screening programme manager Cascade state drives workflow routing, scheduling triggers, and attrition indicators. Without an explicit state model, the system can render individual events but cannot derive aggregate cascade position. Component 4; Indicators — cascade-loss cluster
FR-05 The system SHALL compute the indicators enumerated in Component 8, stratified by the standard disaggregations declared there, and SHALL expose them for programme-manager review and for aggregate export to national health information systems. Screening programme manager Indicator values that cannot be extracted from the system have no programmatic use. Exposure — both human-readable and machine-exportable — is what makes indicator definitions operational. Component 8; NFR-05

FR-06 The system SHALL produce a list of women in the facility catchment who meet the screening-eligibility criteria but have no valid primary-screen record within the applicable interval, AND SHALL flag subsets of that list that have passed the programme-defined overdue threshold. Community health worker The Identify overdue rescreen candidate activity in the Invitation workflow depends on a system-generated list. Primary Screening

ID Requirement Personas served Rationale Cross-refs
FR-08 The system SHALL record an HPV sample collection event with sample date, collecting provider, sample type (clinician-collected or self-collected), and link to the woman's stable patient identifier. Nurse/midwife; Target client (if self-collection) Provides one side of the Primary Screening laboratory message exchange; is the cascade-state-changing event that moves a woman from eligible-not-screened to sample-collected-awaiting-result. Component 4 — Primary Screening; Component 5
FR-09 The system SHALL record an HPV DNA result as positive, negative, or invalid, with result date, laboratory identifier, and an explicit link to the originating sample collection event recorded under FR-08. Laboratory technician The HPV DNA result is the cascade's generative fact — it triggers either routine follow-up (negative) or the triage pathway (positive). It is the input to indicators IND-01, IND-03, IND-04, and IND-06. Component 4; Component 5 Phase-2 gap; Component 8
FR-10 The system SHALL convey HPV DNA results from the laboratory context to the originating primary-care facility context, and SHALL notify the facility when a new result becomes available for a registered woman. Lab tech (sender); Nurse/midwife (receiver) Operationalises the Primary Screening workflow's MessageFlow_ResultReturn. Without an automated result-return mechanism, facilities must poll the lab, which creates routine delays that reduce IND-04 triage attendance. Component 4 — Primary Screening message flows

Triage

ID Requirement Personas served Rationale Cross-refs
FR-11 The system SHALL record a VIA triage encounter with date, performing provider, and result coded as one of: triage-negative, ablation-eligible positive, non-ablation-eligible positive, suspicious for invasive cancer, or inadequate. Nurse/midwife Captures the triage classification that drives the Treatment workflow's disposition gateway and that feeds indicators IND-04, IND-05, IND-09, and IND-10. The five-value result vocabulary is what makes Algorithm 5 operationalise cc-003's suspected-cancer branch and IND-10's adequacy concept. Component 4 — Triage; Component 5; Component 6; cc-003

Treatment

ID Requirement Personas served Rationale Cross-refs
FR-12 The system SHALL support a treatment-eligibility decision that takes as input the triage result (FR-11), the ablation-eligibility lesion description where applicable, and the woman's pregnancy status (FR-15), and SHALL route the subsequent workflow to ablation, excision referral, pregnancy deferral, or suspected-cancer referral accordingly. Nurse/midwife Drives the Treatment workflow's disposition gateway. Without this decision, the cascade cannot differentiate ablation-in-facility from excision-at-district-hospital, nor respect the pregnancy-deferral GPS. Component 4 — Treatment; Component 6; cc-006
FR-13 The system SHALL record a treatment event with date, treatment type (thermal ablation / LLETZ-LEEP / cold-knife conization), performing provider, facility (primary-care or district-hospital), and outcome coded as delivered / deferred / referred-onward. Nurse/midwife; Advanced clinician Drives post-treatment retest scheduling (FR-16) and indicators IND-05 and IND-07. The separation of treatment-type from outcome makes the pregnancy-deferred case and the suspected-cancer-referred case distinguishable in indicator numerators. Component 4 — Treatment; Component 5 Phase-2 gap
FR-14 The system SHALL convey a cross-facility referral from the primary-care facility to the district hospital — for non-ablation-eligible precancer (LLETZ/CKC) and for suspected invasive cancer — carrying sufficient clinical context (triage findings, screening history, contact details) for the receiving advanced clinician to evaluate and act. Nurse/midwife (initiating); Advanced clinician (receiving) Operationalises the two cross-pool message flows in the Treatment workflow. A referral that arrives context-free forces duplicate history-taking, delays care, and produces cascade attrition between FR-13's referred-onward outcome and the actual excision event. Component 4 — Treatment message flows
FR-15 The system SHALL capture the target client's pregnancy status at the treatment-eligibility decision step (FR-12), and SHALL invoke the pregnancy-deferral branch of the disposition gateway when pregnant, routing the workflow to the treatment-deferred-pregnancy cascade state (FR-04) with a scheduled post-pregnancy re-evaluation. Nurse/midwife Operationalises the rec 41 good practice statement on pregnancy deferral, which is the principal purposive-textualist divergence encoded in cc-006. Without pregnancy capture at this step, the cascade would proceed to contraindicated ablation in pregnant women. cc-006; Component 4 — Treatment; Component 6

Follow-up

ID Requirement Personas served Rationale Cross-refs
FR-16 The system SHALL schedule the next encounter for a woman based on her current cascade state and the applicable interval rule — routine rescreening after HPV-negative result, triage-negative retest, or post-treatment retest — and SHALL compute the scheduled date using the interval values owned by cc-005 and cc-002. Nurse/midwife Operationalises the Follow-up workflow's three schedule activities. Hard-coding the interval values in application logic rather than reading them from the interpretation register would create country-adaptation debt; the system must read interval values as configuration, not code. Component 4 — Follow-up; Component 7; FR-20
FR-17 The system SHALL issue a reminder to the target client (by SMS or alternative channel configured per facility) in advance of the scheduled encounter, and SHALL flag the record for CHW outreach when the encounter is not attended within the programme-defined overdue threshold. Target client (recipient); CHW (outreach trigger) Operationalises the three message flows from Follow-up schedule activities to the Client pool. The reminder lead time and overdue threshold are country-adaptation parameters declared in Component 7 and configurable per FR-20. Component 4 — Follow-up; Scheduling Logic — deferred parameters; FR-20

Operational capabilities

ID Requirement Personas served Rationale Cross-refs
FR-19 The system SHALL record an audit trail of all cascade-state-changing events (HPV results posted, triage decisions recorded, treatments delivered, encounters scheduled, cascade-state transitions) with timestamp, actor, and (for updates) the prior value. Screening programme manager; quality-assurance reviewer Supports retrospective quality review, indicator validation, and accountability under the data-protection framework (NFR-07). Audit-trail presence is also a standard requirement of most country health-data-governance frameworks. NFR-07; Component 8
FR-20 The system SHALL provide an administrative interface by which a country programme can configure the country-adaptation parameters named in Components 7 and 8 — at minimum: reminder lead time, overdue threshold, the three screening interval bands (HPV-negative rescreen, triage-negative retest, post-treatment retest), age-band definitions, HIV-status value set, and the treatment-within-6-months tracking convention (rec 41 GPS). Screening programme manager; national IT administrator Operationalises the SOP's 80/20 adaptation boundary and this DAK's explicit declaration of country-adaptation parameters. Without a configuration UI, every adaptation requires code deployment, which is the opposite of the reusability principle this DAK commits to. Scheduling Logic; Indicators; cc-000

Non-functional requirements

Twelve non-functional requirements focused on LMIC-deployment realities — the quality-attribute commitments a digital system must make for Algorithm 5 to actually run in the settings the DAK targets.

ID Requirement Personas served Rationale Cross-refs
NFR-01 The system SHALL support offline operation at the point of care for at least 72 hours of continuous facility activity, with automatic synchronisation to the central record upon restored connectivity. Nurse/midwife; Laboratory technician; CHW LMIC peripheral facilities routinely experience multi-day connectivity outages. Requiring online operation renders the DAK unimplementable at exactly the deployment level Algorithm 5 targets. cc-000; Algorithm Selection
NFR-02 Synchronisation SHALL function over low-bandwidth connections (sustained throughput of 64 kbps or less) and SHALL tolerate connection interruptions, resuming from the last successfully synchronised state without data loss. Nurse/midwife; IT administrator Cellular and satellite connectivity at peripheral facilities is commonly bandwidth-constrained and intermittent. A sync design that assumes broadband will fail in deployment. NFR-01
NFR-03 The system SHALL operate on tablet-class devices (7-inch and larger) for clinical documentation, and SHALL provide client variants (web, native, or USSD) appropriate for CHW outreach on feature-phone-class devices where tablets are not distributed to community health workers. Nurse/midwife (tablet); CHW (feature-phone or tablet) Device provisioning is a major cost driver. A single-device-class assumption either over-provisions or under-supports. Component 2 — CHW persona; NFR-04
NFR-04 The system SHALL present user interfaces in the local language(s) of the deployment country, and SHALL support extension to additional languages without application redeployment. All end-user personas Language adaptation without redeployment is what makes the system deployable across multilingual countries (the ministry cannot wait for a vendor release cycle to add a new provincial language). Component 2
NFR-05 The system SHALL exchange clinical data using FHIR R4 or R5 resources using the data-element semantics defined in the Core Data Dictionary, and SHALL export aggregate indicator data in a form consumable by DHIS2 or equivalent aggregate-reporting systems. IT architect; Screening programme manager; cross-DAK integration consumers Interoperability is the concrete commitment behind the WHO SMART ecosystem. FHIR-based exchange also prepays L3 Measure / CQL authoring for indicators (Phase-2 indicator follow-on). Component 5; Component 8; FR-05; FR-18
NFR-06 The system SHALL encrypt personally identifiable and clinical data at rest and in transit, and SHALL comply with the data-protection framework of the deployment country (or equivalent international standard where no country framework is in force). Target client (data subject); IT administrator Basic data-protection hygiene is both an ethical baseline and increasingly a regulatory requirement. Framing compliance country-by-country avoids over-committing to a single standard while still setting a floor. FR-19
NFR-07 Interactive responses to cascade-state queries and decision-logic evaluations (FR-02, FR-04) SHALL return within 2 seconds under typical facility load; batch indicator computations (FR-05) SHALL return within 30 seconds for a single facility's monthly cohort. Nurse/midwife; Screening programme manager Point-of-care decision support that takes 10 seconds will be bypassed; programme-manager indicator review that takes 10 minutes will be delayed. Interactive and batch thresholds need to be stated separately. FR-02; FR-04; FR-05
NFR-08 System availability at the point of care during facility operating hours SHALL be at least 99.0% aggregating local-client availability and the offline mode specified in NFR-01. A nurse unable to document an encounter is a scheduling and indicator-integrity incident. Nurse/midwife; Screening programme manager The 99% floor is softer than typical SaaS SLAs because it already accounts for offline operation. Without the floor, intermittent-system-outage excuses begin to compete with clinical work. NFR-01
NFR-09 A trained nurse/midwife SHALL be able to complete cascade documentation for a typical encounter (primary screen or triage) within 5 minutes. Nurse/midwife Documentation time is a direct cost on clinical throughput. A system that extends encounter time materially reduces daily screening throughput and defeats the 70% coverage target. Component 2 — Nurse/midwife persona; Indicators — IND-01
NFR-10 The system SHALL provide context-sensitive help and onboarding materials appropriate to each persona — CHW, nurse/midwife, laboratory technician, advanced clinician, screening programme manager. All end-user personas New-hire onboarding is continuous in LMIC health workforces. The system must support self-paced learning without requiring in-person training for every cohort. Component 2
NFR-11 The system SHALL permit migration between cervical cancer screening algorithms (most plausibly from Algorithm 5 to Algorithm 4 as HPV DNA platforms add genotyping) without requiring re-authorisation of existing patient records or loss of historical cascade state. IT architect; Screening programme manager The Algorithm Selection analysis explicitly identifies Algorithm 4 as the most likely future migration target. A system that hard-codes Algorithm 5 into its data model forecloses that migration path and creates long-term technical debt. Algorithm Selection; FR-20
NFR-12 The system SHALL support bulk data import of existing patient records from predecessor systems (paper registers digitised; predecessor EHR module; predecessor VIA-programme tracker), preserving historical cascade events where recoverable. Screening programme manager; IT administrator Few countries deploy a DAK-conformant system greenfield. Bulk-import support converts the pre-DAK patient record base into a starting corpus rather than a discarded history. NFR-06; FR-03

Relation to other components

  • Component 1 — Recs 41 (GPS on pregnancy deferral) grounds FR-15; rec 3b / 23 (triage method delegation) grounds the Algorithm-5 specificity of FR-11 and FR-14.
  • Component 2 — Every functional requirement names at least one persona from the generic persona set. NFR-03, NFR-04, NFR-09, NFR-10 are particularly persona-driven.
  • Component 3 — The seven user scenarios are the demonstration set against which these requirements should be walked end-to-end; a requirement gap is most visible when a scenario cannot be narrated against a candidate system.
  • Component 4 — The five BPMN workflows plus the counselling sub-process determine the cluster structure of the functional requirements.
  • Component 5 — FR-03 is the blanket commitment that every CDD element is captureable. The Phase-2 CDD gaps flagged on the Indicators page are requirements on the CDD, not on the system implementing FR-03.
  • Component 6 — FR-02 is the blanket commitment that every decision table is evaluable. Future DMN additions for triage-negative and post-treatment retest scheduling (flagged Phase-2 in cc-005 and Scheduling Logic) inherit FR-02 by extension.
  • Component 7 — FR-16, FR-17, and FR-20 are the scheduling-logic-facing capabilities. The three country-adaptation parameters named in Component 7 are enumerated concretely in FR-20.
  • Component 8 — FR-05, FR-18, NFR-05 are the indicator-computation-and-export chain. The Tier A / Tier B feasibility split on Component 8 maps directly onto whether a requirement's dependencies are already in the CDD (Tier A) or pending (Tier B).

Status

Draft — Phase 1 of Component 9 authoring (2026-04-22). Twenty functional requirements organised by cascade module plus a cross-cutting cluster; twelve non-functional requirements focused on LMIC-deployment realities. Each requirement carries a statement, served personas, rationale, and cross-references back to the originating component(s). SME validation — from a national or regional digital health architect with SMART DAK implementation experience — is deferred with the same stratified pattern applied to earlier components.

Phase-2 follow-on work for this component:

  • Implementer / vendor review. Requirements of this form benefit from review by someone who has actually configured or built an EHR or screening registry; blind spots in Phase-1 analytical derivation (particularly in NFRs on performance, offline sync, and bulk import) often only surface against lived implementation experience.
  • Test-case derivation. Each functional requirement corresponds to one or more acceptance criteria / test cases that a procurement or conformance exercise would use. Those test cases are a Phase-2 artifact that the Component 3 user scenarios naturally feed into.
  • L3 conformance test suite. Once CDD Phase-2 work closes the data-element gaps and L3 Measure / CQL artifacts are authored, the requirements on this page become an input to a conformance test suite against which candidate implementing systems can be evaluated.
  • Cross-DAK harmonisation. Many of these requirements — especially the cross-cutting and NFR sets — are generic across WHO SMART DAKs. A shared canonical-requirements catalogue (analogous to the shared-dictionary aspiration of the Data Dictionary README) would reduce per-DAK authoring burden and improve cross-DAK consistency.
